Introduction

Un bon outil de versionnage va nous transformer la vie de développeur pour plusieurs raisons:

  • On a un historique de tous les changements réalisés dans le code
  • On peut travailler sur des branches différentes
  • On peut travailler en collaboration avec d'autres développeurs

La première raison est celle qui nous intéresse le plus et pour bien la comprendre, on va se plonger dans une situation. On veut écrire du code dans le but de créer un réveil interactif qui réagit au son de la voix (Siri en moins bien quoi). Au bout de quelques temps, on a un code à peu près stable qui permet de détecter l'ordre "Fermes là!" pour que le réveil arrête l'alarme. A partir de ce moment là, on va généralement venir faire une sauvegarde de son travail quelque part sur son disque dur ou sur un cloud. Jusque là pas de soucis. Les mois avancent et des nouvelles features viennent s'ajouter au code initial. Parfois on fait des sauvegarde après une feature, parfois on en fait pas par flemme ou par oubli. Et le jour du drame arrive. Un utilisateur de notre programme nous fait la bonté de remonter un bug qu'il a rencontré en nous donnant les étapes à reproduire. Alors on comment à reproduire, on essaye de se plonger dans le code pour voir d'où vient le problème, mais on ne comprend rien de ce qu'il se passe, c'est la tristesse et là, plus le choix, on doit revenir sur les sauvegardes précédentes de notre code pour trouver la source du bug. On commence à prendre une sauvegarde assez ancienne, à le décompresser, à le recompiler, à refaire les étapes et à tester si le bug est toujours là. On refait çà des dizaines de fois jusqu'à tomber sur la bonne sauvegarde. Et là regalère, parce qu'il y a énormément de changement entre le sauvegarde qui fonctionne bien et la prochaine sauvegarde qui créé le bug. Au final on est pas aidé plus que çà, il faut se replonger dans le code et comprendre ce qu'il se passe. On voit bien ici que c'est long et fastidieux d'aller chercher les sauvegardes, pour les essayer une par une. Imaginez qu'en plus ces sauvegardes sont assez volumineuses, et pire! que ces sauvegardes sont sur un cloud ou encore PIRE! que vous avez supprimé vos anciennes sauvegardes parce que çà prenait trop de place. Bref, c'est la galère!

Premier problème c'est la place que prend une sauvegarde. On peut se retrouver rapidement dans une situation où çà prend plusieurs go de mémoire. L'utilisation du cloud peut être une solution mais au dépit d'un temps à rallonge à chaque upload et download (dans le cas d'investigation d'un bug).

Second problème, c'est le fait de devoir recompiler la totalité du code qui peut prendre parfois plusieurs minutes à parfois plusieurs dizaines de minutes à cause des copies systématiques des dépendances notamment.

Troisième problème, qui est lié au premier, c'est la fréquence de sauvegardes que l'on fait. On peut être très rigoureux et faire une sauvegarde après chaque feature et/ou bug corrigé, voire même après chaque jour de travail, ou chaque demi-journée. Mais on se retrouve avec un serpent qui se mord la queue, puisque l'on multiplie le nombre de sauvegarde et donc le temps d'investigation pour un bug.

Quatrième problème, c'est la difficulté de faire la différence fichier par fichier entre deux sauvegardes. On peut utiliser des logiciels qui sont dédiés à çà comme beyond compare, mais c'est pas super automatique, et là aussi on perd du temps.

Je n'ai même pas évoqué l'infaisabilité des deux autres raisons évoquées au tout début comme le travail d'équipe où on voit souvent des situations avec des étudiants qui s'échangent des clé USB avec le code dessus et qui s'engueulent parce que l'un des deux a modifié le code de l'autre et qu'il se retrouve avec plein de conflits et donc des bugs. Ou alors lorsque qu'on veut faire un peu d'expérimentation.

Bref, ce serait quand même super si quelqu'un avait une solution pour çà. Et là, il y a quelqu'un qui est arrivé et qui a fait : "Hey mais les gars, au lieu de s'embêter à faire des photos (sauvegardes) de notre code à chaque fois, est-ce qu'on ferait pas plutôt des photos des changements? Je vous promet que çà va nous changer la vie!". Celui qui a imaginé Git (Linus Torvalds, le papa de Linux) a sauvé la vie de la quasi totalité des développeurs et je vais vous expliquer pourquoi. Disclaimer : Alors git n'est pas le premier outil de versionnage, mais on peut quand même dire que c'est l'outil qui a très largement popularisé, normalisé et convergé les pratiques. De plus, Mr Linus n'a pas été seul dans sa tâche, puisque de nombreux développeurs ont contribué au projet pour avoir un outil aujourd'hui utilisé par plus de 10 millions de développeurs.

Git, un outil de versionnage, quesaco?

Git est un outil permettant d'enregistrer les changements réalisées entre deux versions de code. Prenons par exemple un fichier source qui se présente de la façon suivante :

#include <iostream>
int main()
{
    std::cout << "Hello world" << std::endl;
    return 0;
}

 

Admettons que je change ce code en ajoutant une ligne comme le code suivant:

xxxxxxxxxx
#include <iostream>
int main()
{
    std::cout << "Hello world" << std::endl;
    std::cout << "Hello you!" << std::endl;
    return 0;
}

 

Si je demande à Git la différence entre les deux fichiers, il m'affichera :

xxxxxxxxxx
diff --git a/main.cpp b/main.cpp
index 29de88e..00f7fab 100644
--- a/main.cpp
+++ b/main.cpp
@@ -3,5 +3,6 @@
 int main()
 {
        std::cout << "Hello world" << std::endl;
+       std::cout << "Hello you!" << std::endl;
        return 0;
 }
\ No newline at end of file

 

On voit clairement que la différence affichée entre des deux fichiers est bien l'ajout de notre nouvelle ligne de code.
