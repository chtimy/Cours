Pointeurs et références (partie 1)

Introduction

C'est parti, commençons cette fiche comme à notre habitude, c'est à dire avec une petite problématique.

L'idée du code est de pouvoir peindre des voiture en passant par une classe dédiée à cette action qui est la classe AtelierPeinture. On envoie alors à l'atelierPeinture une voiture et une couleur pour la peindre avec la couleur demandée.

Si tu as exécuté le code, tu as remarqué que la couleur de la voiture ne change pas. C'est est un gros problème puisque ce n'est pas ce qui est attendu. On voulait que notre voiture créée dans la fonction main change de couleur du rouge vers le bleu après l'avoir envoyé à l'objet atelierPeinture. Que s'est t-il passé?

Lorsque la voiture est envoyée à la fonction peindre, cette voiture est copiée dans le paramètre d'entrée voiture de la fonction. Lorsqu'on modifie alors la voiture dans la fonction peindre, on ne modifie que la copie de la voiture de la fonction main. Les deux objets Voiture vivent de manière indépendante. On ne peut donc jamais modifier la voiture de la fonction main en faisant de cette façon, il va falloir trouver un autre moyen.

Mais on pourrait pas juste retourner la voiture dans la fonction peindre?

On peut totalement, problème réglé, on arrête la fiche dès maintenant... Et bien non :) c'est pas aussi simple. Je vais d'abord reproposer le code avec cette solution.

Il était une fois, un facteur...

Dans cette partie, je vais tenter une métaphore comme préambule pour tenter d'expliquer à la fois le comportement de la mémoire, mais aussi ce que signifie variable et pointeur. Si jamais ma métaphore ne fonctionne pas, et bien... restes et lis la suite de cette fiche où je vais faire plein de dessins!!

Et voici ma métaphore. On peut imaginer la mémoire où chaque case est une boite aux lettres. Chaque boîte aux lettres a une adresse, sinon le facteur aurait du mal à la trouver. Une boîte aux lettres peut contenir quelque chose. Comme l'adresse d'une boîte aux lettres c'est souvent long à lire et pas super facile à retenir, on lui donne alors un petit nom comme "Ma maison", c'est ce que sont les variables : des alias d'adresses. Lorsque tu fais int a;, a est un nom alias d'une adresse en mémoire, c'est juste que tu ne le vois pas, et c'est bien plus facile comme çà que de manier une adresse.

On peut demander au facteur de mettre quelque chose dans une boîte aux lettres. Ce quelque chose peut être un entier, un flottant, un caractère, un objet, ... (il faut penser un peu de manière méta pour imaginer çà, mais j'espère que tu vois ce que je veux dire). C'est ce que tu fais avec par exemple int a = 5;.

Imaginons maintenant qu'au lieu de mettre un quelque chose dans la boite au lettre, on y mettrait un bout de papier indiquant l'adresse d'une autre boite aux lettres. Ce genre de boîte aux lettres est ce qu'on appelle un pointeur. C'est une variable où on stocke une adresse.

On va maintenant demander au facteur de mettre quelque chose à une adresse que je ne connais pas, mais dont je sais qu'elle est stockée dans une autre boite aux lettres (le pointeur). Ce que le facteur va faire, c'est aller à la boite aux lettre contenant l'adresse, l'ouvrir et lire l'adresse qui est à l'intérieur. A partir de cette adresse, le facteur peut retrouver la boîte aux lettre finale et ainsi déposer le quelque chose dedans.

La boîte aux lettres contenant une adresse est donc ce qu'on appelle un pointeur et elle référence peut référencer une autre boite aux lettres. Le fait que le facteur puisse accéder à la boîte aux lettres finale à partir de la boîte aux lettres "pointeur", c'est ce qu'on appelle le déréférencement (ou l'indirection).

Les variables : des alias d'adresse mémoire

Je tente dans cette partie, d'expliquer la notion de pointeur en reprenant tout de zéro. Commençons par la notion d'adresse mémoire en partant d'un petit exemple: 

x
int main()
{
    int a = 5;
    int b = 3;
    b = a;  
    a = 6; 
    std::cout << a << std::endl;
    std::cout << b << std::endl;
}

Quelles sont les valeurs de a et b?

A la ligne b = a, la valeur stockée dans a est copiée dans la variable b, donc b vaut 5. A la ligne a = 6, la valeur 6 est affectée à la variable a, ce qui signifie qu'à la fin a vaut 6 et b vaut 5.

C'est important de comprendre ce qu'il se passe ici en mémoire pour ne pas être perdu dans la suite de cette fiche. On va représenter la mémoire de la façon suivante :

Je met ici le caractère '@' pour spécifier que c'est l'adresse d'une case mémoire. La première case de la mémoire a pour adresse @0. Chaque case mémoire contient une valeur. Pour simplifier les choses ici, on considère que chaque case mémoire peut contenir un type quelconque comme un entier.

Reprenons l'exemple de code précédent et regardons ce qu'il se passe en mémoire étape par étape.

Etape 1

a = 5;

Le programme va réserver une case mémoire à l'adresse &a et écrire la valeur 5 dans cette case mémoire. L'adresse @a est une adresse dont on a pas besoin de connaitre, et c'est ce qui rend l'utilisation des variables indispensables pour la programmation, puisque toutes ces étapes de réservation, écriture et accès mémoire se font à travers le nom des variables. Ce qu'il faut retenir ici donc, c'est que la variable a correspond à la valeur stockée dans la case mémoire à l'adresse @a, c'est à dire 5.

Etape 2

b = 3;

C'est la même chose pour la variable b. On lui affecte la valeur 3 dans la case mémoire qui lui a été réservée à l'adresse @b.

Etape 4

xxxxxxxxxx
b = a;

Cette ligne signifie que le programme va accéder à la valeur à l'adresse @b pour venir la copier dans la case mémoire à l'adresse @a. Les deux cases @a et @b ont la même valeur 5.

Etape 5

xxxxxxxxxx
a = 6;

Dans ce cas, la valeur 6 remplace l'ancienne valeur de la case mémoire à l'adresse @a.

On vient de dérouler tout l'algorithme de cette partie pour bien mettre l'accent sur ce qu'il se passe en mémoire et que des copies sont réalisées lorsqu'une variable est affectée à une autre variable. Toutes les variables qui sont déclarées dans le code sont obligatoirement des alias d'une adresse mémoire.

Les pointeurs : das variables presque comme les autres

A ce stade, tu sais que les variables ont un type spécifique. On peut citer des types génériques comme :

  • int pour les entiers
  • float pour les flottants
  • char pour les caractères
  • ...

Il existe aussi d'autres types de variable comme les objets qui représentent des classes ou des structures. Exemple: 

x
class A
{
    private:
    int _value;
    public:
    A (value)
    {
        _value = value;
    }
};
struct B
{
    int value;
};
void main
{
    A a(5); // la variable a est de type A
    B b{5}; // la variable b est de type B    
}

Et nous voilà aux pointeurs qui représentent encore un autre type de variable. Un pointeur est une variable contenant l'adresse mémoire d'une autre case mémoire. Avant d'en dire plus, je vais donner un exemple de code. 

x
void main
{
    int a = 5;
    int* b = &a;
}

Dans cet exemple, on a créé une première variable a de type int avec pour valeur 5. Comme tu le sais maintenant, en mémoire ce serait donc comme çà:

Ensuite vient ce qui nous intéresse : la variable b. Cette variable est de type int* et là deux choses importantes à comprendre ici. Lors de la déclaration d'une variable, lorsque le type contient une *, çà signifie que la variable est un pointeur et plus précisément que c'est une variable qui contient une adresse d'une autre case mémoire.

On ne peut donc affecter à b qu'une adresse mémoire. C'est ce qu'il se passe dans le code en donnant l'adresse de la variable a en utilisant un opérateur spécial représenté par le symbole &. Le choix du symbole est complètement arbitraire en c++. Le fait de venir chercher l'adresse d'une variable avec ce symbole &, c'est ce qu'on appelle le référencement de la variable a. Le pointeur b contient maintenant l'adresse de la variable a.

Si on écrivait a sans le &, on accèderait seulement à la valeur de a, pas son adresse, attention de bien faire la différence.

Déréférencement d'un pointeur

On vient de voir qu'une variable pouvait contenir l'adresse d'une autre variable. Maintenant on va voir comment, à partir d'un pointeur, on peut accéder à la valeur de la variable pointée.

L'opération d'accès à la valeur d'une variable pointée est appelée déréférencement (ou aussi indirection) par opposition au référencement que l'on vient de voir plus haut. Dans le code, çà se traduit par : 

x
void main
{
    int a = 5;
    int* b = &a;
    int c = *b;
}

A la ligne int c = *b;, il se passe deux étapes. La première c'est l'opération de déréférencement qui est réalisée à droite du = lorsqu'on écrit *b où l'idée est de venir chercher la valeur dans l'adresse mémoire pointée par b, c'est à dire @a (étapes 1a et 1b dans l'image ci-dessous). C'est l'équivalent d'écrire int c = *(&a); ou bien de manière simplifiée int c = a;. Ce qui nous amène à la seconde étape que tu connais très bien, c'est à dire que la valeur à droite du = est copiée dans la variable à gauche du =, c'est à dire c (étape 2 dans l'image ci-dessous).

Qu'est-ce qu'il se passe si on met l'opération de déréférencement à gauche du =, comme par exemple *b = 3;

En effet c'est l'autre cas que l'on peut rencontrer. Le comportement n'est pas si différent, puisque qu'on à dit qu'écrire *b, c'est l'équivalent d'écrire *(&a) ou a. Si on fait l'exercice ici, on aurait a = 3 et qu'est-ce qu'il se passe ici?

Euhhh bah, la valeur 3 est copiée dans la variable a, c'est çà?

Exactement! On a pu modifier la variable a par l'intermédiaire du pointeur b. Je refais un petit dessin pour bien se visualiser:

Attention à bien faire la différence entre la déclaration d'un pointeur et le déréférencement d'un pointeur, puisque les deux s'écrivent de la même façon avec l'étoile avant le nom de la variable mais ont des comportements différents. int *a; ou int* a; est une déclaration de pointeur, tandis que int b = *a; est un déréférencement du pointeur a.

Ok, super, mais çà nous avance à quoi tout çà?

On arrive presque au but. Rappelles toi que la problématique initiale est comment envoyer un objet à une fonction sans le copier et pouvoir le modifier directement dans cette fonction.

Envoi d'un pointeur à une fonction

Dans cette partie je rétro pédale, et je vais me répéter, mais je pense que c'est bien de le faire pour pas louper un rouage.

Partons d'un cas simple: 

x
#include <string>
#include <iostream>
class Path
{
    public:
    Path(std::string v)
    {
        value = v;
    }
    std::string value;
};
void toUnixPath(Path path)
{
    for(int i = 0; i < path.value.size(); i++)
    {
        if(path.value[i] == '\\')
        {
            path.value[i] = '/';
        }
    }
}
int main()
{
    Path windowsPath("c:\\users\\superprogrammer");
    toUnixPath(windowsPath);
    std::cout << windowsPath.value << std::endl; // Affiche "c:\users\superprogrammer"
    return 0;
}

L'idée du bout de code ci-dessus est de transformer un chemin "Windows" en chemin "Unix". Pour çà il faut automatiquement remplacer les caractère \ par le caractère /.

Alors si vous lisez bien le code au dessus, lorsque l'objet windowsPath est envoyée à la fonction toUnixPath, il est copié dans la variable path qui est le paramètre d'entrée de la fonction. La transformation du chemin se fera donc sur l'objet copié et non sur l'original. L'objet copié, lui, est bien transformé, mais est détruit lorsqu'il a atteint la fin de la fonction puisqu'il n'est pas renvoyé. Pour le moment le code que l'on a écrit ne fonctionne pas puisqu'on ne récupère pas le chemin transformé.

Une solution serait de renvoyer le chemin transformé afin de le récupérer dans la fonction main comme suit: 

x
#include <string>
#include <iostream>
class Path
{
    public:
    Path(std::string v)
    {
        value = v;
    }
    std::string value;
};
Path toUnixPath(Path path)
{
    for(int i = 0; i < path.value.size(); i++)
    {
        if(path.value[i] == '\\')
        {
            path.value[i] = '/';
        }
    }
    return path;
}
int main()
{
    Path windowsPath("c:\\users\\superprogrammer");
    windowsPath = toUnixPath(windowsPath);
    std::cout << windowsPath.value << std::endl; // Affiche "c:/users/superprogrammer"
    return 0;
}

Ce code fonctionne parfaitement, et il pourrait même être considéré comme bien écrit, mais le problème vient plutôt du côté des performances. Si on déroule un peu l'appel de la fonction, rappelles-toi, on a une première copie lorsqu'on envoie le chemin à la fonction ToUnixPath. Ensuite la copie est transformée dans la fonction pour finalement être renvoyée. L'instruction return va aussi faire une copie du chemin path pour venir finalement l'écrire dans la variable windowsPath dans la fonction main. On a donc deux copies d'un même chemin. Dans le cas où le chemin fait par exemple 5000 caractères, çà veut dire copier deux fois les 5000 caractères, soit 10000 opérations de copies. Ce n'est pas vraiment une grosse difficulté pour les ordinateurs d'aujourd'hui, c'est vrai, mais si on fait çà à chaque fois pour plein d'autres fonctions, çà peut devenir assez problématique. Donc autant bien coder dès le départ.

Le but ici est de ne faire aucune copie du chemin et c'est justement à ce moment là qu'interviennent les pointeurs pour nous sauver. On vient de voir qu'un pointeur était une variable contenant une adresse mémoire et qu'avec l'opération de déréférencement, on pouvait accéder à la valeur stockée à l'adresse pointée. Et bien dans ce cas, au lieu de copier l'ensemble du chemin windowsPath, on pourrait ne copier que son adresse et l'envoyer à la fonction. Ensuite, dans la fonction toUnixPath, on pourrait accéder au même chemin avec l'opération de déréférencement. Bon allez, le code parle toujours mieux de lui-même. 

x
#include <string>
#include <iostream>
class Path
{
    public:
    Path(std::string v)
    {
        value = v;
    }
    std::string value;
};
void toUnixPath(Path* path)
{
    for(int i = 0; i < (*path).value.size(); i++)
    {
        if((*path).value[i] == '\\')
        {
            (*path).value[i] = '/';
        }
    }
}
int main()
{
    Path windowsPath("c:\\users\\superprogrammer");
    toUnixPath(&windowsPath);
    std::cout << windowsPath.value << std::endl; // Affiche "c:/users/superprogrammer"
    return 0;
}

Si tu observes le paramètre d'entrée de la fonction toUnixPath, on a remplacé le type du paramètre de Path avec Path*. Ce paramètre attend donc une adresse d'une case mémoire contenant une valeur de type Path.

Lorsqu'on regarde l'intérieur de cette fonction, on peut voir que l'on déréférence la variable path soit pour accéder à la taille du chemin dans la boucle (i < (*path).value.size()) ou bien pour modifier directement ce chemin à l'intérieur de la boucle ((*path).value[i] = '/';. Maintenant il faut savoir ce qu'il se passe à l'appel de cette fonction.

Je fais un petit point sur la syntaxe un peu bizarre qu'est (*path).value. Rappelles-toi, que pour accéder à l'attribut d'un objet, on fait cette opération objet.attribut, mais comme là on manipule un pointeur sur un objet, alors il faut d'abord déréférencer le pointeur (pour accéder à l'objet) pour ensuite accéder à l'attribut de l'objet (*pointeurSurObjet).attribut

Dans la fonction main, on voit que quelque chose a changé au moment de l'appel de ToUnixPath, puisqu'au lieu d'envoyer le chemin lui-même, on envoie son adresse en utilisant l'opérateur de référencement &. Durant l'exécution du programme, on aura donc la variable windowsPath et la variable path (le paramètre de la fonction) qui accèderont au même chemin.

Dans l'exemple de code précédent, on faisait deux copies du même chemin. Dans cet exemple de code, on en fait aucune. La seule copie qu'il y a, c'est la copie de l'adresse du chemin.

J'aimerai revenir rapidement sur la syntaxe, parce qu'elle me plait pas et qu'on peut faire autrement. Si tu as remarqué, j'ai mis l'opération de déréférencement entre parenthèses comme par exemple dans l'instruction (*path).value[i] = '/';. C'est parce que si je ne le met pas, le compilateur va croire qu'on fait çà : *(path.value[i]) et va déréférencer l'attribut value[i] qui causera une erreur de compilation puisque ce n'est pas un pointeur. Dans tous les cas, on peut écrire de manière bien plus simple, voilà ce qu'ont proposé les ptits gars du c++. A la place d'écrire (*objet).attribut, on peut écrire directement avec cette syntaxe là : objet->attribut. C'est plus clair non?

La seule chose qu'il faut bien retenir, c'est que objet dans objet.attribut n'est pas un pointeur, alors que objet dans objet->attribut est un pointeur. Je met bien les gros warning ici pour faire l'effort de s'en souvenir, parce qu'on peut vite se tromper lorsqu'on a pas l'habitude.

Le pointeur doit démasquer sa cible!

Je vais maintenant revenir sur la remarque du pourquoi on doit spécifier le type de la valeur pointée lors de la déclaration d'un pointeur. Il existe au moins deux raisons :

  • C'est nécessaire pour le déréférencement (lecture de la valeur pointée) d'un pointeur,
  • C'est nécessaire à cause de l'arithmétique sur les pointeurs

La première raison vient du fait que chaque type n'a pas la même taille. Prenons par exemple le type char qui a une taille de 1 octet (8 bits) et le type int qui a en général une taille de 4 octets (32 bits). Le déréférencement d'un pointeur pointant une valeur de type int serait de lire le nombre d'octets correspondant au type int, soit 4 octets, afin de décoder correctement la valeur pointée. Si la valeur pointée est un int et que le programme le lit comme un char, il ne va lire qu'un seul octet au lieu de 4 et donc on aura pas toutes les informations nécessaires pour constituer un int. A l'inverse, si la valeur pointée est un char et que le programme le lit comme un int, il va lire bien trop d'octets et dépasser sur de la mémoire qui ne le concerne pas et çà va devenir n'importe quoi. En bref, il faut indiquer au pointeur le type de la valeur pointée afin qu'il ne se trompe pas de type lors du déréférencement.

La seconde raison est un peu plus complexe à expliquer dès maintenant, mais j'essaye. Prenons pas exemple un pointeur sur un type int. Si je fais la chose suivante : pointeur++, je n'ajoute pas 1 à l'adresse stockée dans le pointeur mais j'ajoute la taille d'un type int, c'est à dire que le pointeur va lire 4 octets plus loin, soit le prochain int probablement. A l'inverse pointeur-- va lui lire 4 octets avant. Si le type de la valeur pointée n'est pas connu par le pointeur, alors il ne sait pas combien d'octets il faut sauter pour aller à la prochaine valeur pointée. Alors pourquoi on fait çà? Ca peut être le cas quand on parcourt un tableau par exemple pour aller d'une case à une autre, mais c'est de moins en moins vrai en c++, on utilisera d'autres mécanisme bien plus haut niveau.

class Voiture

    void setCouleur(int couleur)
    {
        if(couleur < 0)
        {

        }
        _couleur = couleur;
    }
    int getCouleur()
    {
        return _couleur;
    }

class AtelierPeinture

    public:

Résumé

  • La création d'une variable implique la réservation d'une case mémoire ayant une adresse et une valeur,
  • Le nom d'une variable est l'alias de l'adresse de la case mémoire,
  • Lorsqu'une variable est affectée à une autre variable (de même type obligatoirement), alors in y a une copie de la valeur contenue dans la variable à droite du = dans la variable à gauche du =
  • Un pointeur est une variable qui contient l'adresse d'une case mémoire
  • La déclaration d'un pointeur est réalisée de la façon suivante : [Type pointée]* [nom_pointeur]
  • L'opération de référencement consiste à donner l'adresse d'une variable, on l'écrit de cette manière : &[nom variable]
  • L'opération de déréférencement consiste à accéder à la valeur d'une adresse contenu dans un pointeur pour la lire ou pour l'écraser en écriture. L'opération s'écrit de la façon suivante : *pointeur.
  • On peut éviter la copie de gros objets en utilisant des pointeurs. C'est le cas lorsqu'on envoie l'adresse d'une variable à une fonction ayant un paramètre d'entrée de type pointeur.

int main()
{
    int a  = 5; 
    int b = a; // Copie de la valeur de a dans la variable b
    int *c = &b; // copie de l'adresse de b dans le pointeur c
    std::cout << c << std::endl; // Affiche l'adresse de c
    std::cout << *c << std::endl; // Affiche la valeur pointée par c (lecture)
    *c = 3; // Change la valeur pointée par c par 3, ce qui change en conséquence la valeur de b
    b = 1; // Change la valeur de b par 1, ce qui change en conséquence la valeur de *c
    
    fonctionParCopie(a); // Copie la valeur de a dans le paramètre param
    fonctionParPointeur(&a); // Copie de l'adresse de a dans le paramètre param
    fonctionParPointeur(c); // Copie de l'adresse contenue dans c (soit l'adresse de b) dans le pramètre param
    
    return 0;
